---

## 0. 一行で

> **Bezier — ハンドルを握る。曲線はエージェントが描く。**
> *Hold the handles. The agent draws the curve.*

プロダクトデザイナー & PdM が、AI エージェントに本物のソフトウェアを作らせる場所。
デザイナーは少数の **制御点（intent・注釈・taste）** を置く。エージェントが **滑らかな曲線（実装）** を描く。

---

## 1. なぜ「Bezier」なのか — 名前が thesis そのもの

### 1.1 Bézier 曲線とは
ベジェ曲線は **ピエール・ベジェ**（ルノーの技術者）が 1960 年代に車体の曲面設計のために定式化した。
**工学（数式・コード）と視覚的フォルムが交わる、まさにその一点で生まれた**。

- 曲線は、デザイナーがドラッグする **少数の制御点（control points）とハンドル** で定義される。
- 制御点は離散的な「数値・コード」。レンダリングされる曲線は連続的な「視覚」。
- **ペンツールのアンカーとハンドル** — すべてのデザイナーが毎日触っている、最も手に馴染んだ概念。

### 1.2 それが、このプロダクトの構造と完全一致する
`coevolution-positioning` の核：**「continuum はエンジンを持たず、ハンドルとログブックを握る」**。

| ベジェ曲線 | Bezier プロダクト |
|---|---|
| 制御点・ハンドル（人が掴む） | **intent / 注釈 / taste**（デザイナー & PdM が握る） |
| 滑らかな曲線（自動で描かれる） | **実装コード**（エージェントが描く） |
| 少数の点で全体が決まる | 少数の正しい判断で、プロダクト全体が決まる |
| 数値 ⇄ 視覚の橋 | コード ⇄ デザインの橋 |

→ **名前を聞いた瞬間、プロダクトの構造が伝わる**。しかもデザイナーには初日から「自分の言葉」。
これ以上の適合はない。`continuum`（連続体・抽象的）から `Bezier`（具体的・手に馴染む・thesis を内包）へ。

### 1.3 名前が同時に解く3つのブランド要件
- ✅ **デザイナーの手に馴染む** — ペンツールの言語。Figma/Linear/Notion の隣に置いても違和感がない。
- ✅ **terminal の黒い画面への恐れを溶かす** — あなたは制御点を置くだけ。コマンドは打たない。コードはエージェントが描く曲線。§4。
- ✅ **orchestrator 文化にも刺さる** — 「制御点＝あなたの介入点、曲線＝エージェントの実行」は cursor/cmux/superset 民にも一発で通る抽象。

## 2. ポジショニング

### 2.1 ポジショニングステートメント
> **Bezier は、プロダクトデザイナー & PdM のためのエージェント・ワークベンチ。**
> 彼らが「意図を入れて、判断を出す」ことに専念し、実装はエージェントに描かせる。
> Figma があなたとピクセルの間にあるように、**Bezier はあなたとエージェントの間にある**。

### 2.2 カテゴリの置き場所
- ❌ 「AI コードジェネレータ」ではない（生成は誰でも作る、コモディティ層）。
- ❌ 「もう一つのターミナル UI」でもない（orchestrator は便利だがエンジニア向け）。
- ✅ **「デザイナー & PdM を主役に、AI 開発ループを回す唯一のワークベンチ」**。
  - エンジニア中心の Cursor/Claude Code はデザイナーを二級市民にしている。Bezier はそこを一級にする楔。

### 2.3 競合との距離（ブランド体験上の差）
| | 体験の核 | Bezier との関係 |
|---|---|---|
| **Figma** | 直接ピクセルを操る | 同じ「手に馴染む」品質基準を借りる。Bezier は対象がピクセルでなく**動くプロダクト** |
| **Linear** | 静かで速い・意見のある OS | 静謐さ・速度・断固たる意見を継ぐ。Linear が課題を、Bezier が**実装ループ**を扱う |
| **Notion** | 柔らかく everything | ドキュメント＝決定の住処という発想を借りる。Bezier の docs は**実コードに接地** |
| **Cursor / cmux / Superset / kiro** | エージェント orchestration（エンジニア向け） | 同じ抽象を共有しつつ、**視覚的注釈・taste・プロダクト整合**を一級にして差別化 |

## 3. ブランドパーソナリティ

**Precise, but warm.（精緻、しかし温かい）**

ベジェ曲線そのものの性格：数学的に厳密でありながら、出てくる形は有機的で美しい。

| 軸 | Bezier は | Bezier は〜ない |
|---|---|---|
| トーン | 静か・自信・職人的 | うるさい・誇張・hype |
| 知性 | 明晰・本質的 | 賢ぶる・難解 |
| 温度 | 招き入れる・人間的 | クール過ぎる・冷たい |
| 態度 | あなたが主導権を持つ | AI が主役・あなたを置き去る |
| 美学 | 融けて自然（"融けるデザイン"） | 奇抜・装飾過多・トレンド追従 |

ひとことで：**「賢い同僚のデザイナーが作った、静かで気持ちのいい道具」**。

## 4. 戦略メッセージ — 「黒い画面」を溶かす

デザイナーは terminal を苦手に感じがち。Bezier はそれを **隠さず、リフレーミングする**。

### 4.1 リフレーム
- ❌ 「ターミナルを隠して、デザイナーに優しいフリをする」→ 嘘っぽい・いずれバレる。
- ✅ **「コマンドは打たない。あなたは制御点（注釈・intent）を置くだけ。エージェントの実行は『曲線が描かれていく様子』として静かに見える」**。

### 4.2 言い換えの辞書（コピー全体で一貫させる）
| 避ける語（恐れを呼ぶ） | 使う語（手に馴染む） |
|---|---|
| ターミナル / コマンド / CLI | エージェントの作業 / 実行の流れ |
| 実行する / run | 描く / 進める |
| プロンプトを書く | 制御点を置く / 意図を渡す |
| デプロイ・ビルドエラー | （見せるが、静かなステータスとして。脅さない） |
| コードを書く | 形にする |

### 4.3 見え方への要求
- エージェントの作業ログは **黒いコンソール調にしない**。背景は作業面と地続き、文字は穏やかな階調、進行は「流れ」として読める。
- ただし **隠さない**（透明性は信頼の源）。「見えるが、要求されない」。あなたはいつでもハンドルを握っている。
- 唯一の例外：明示的に terminal を開きたい上級者には、ちゃんとした端末を渡す（恐れさせないが、奪わない）。

## 5. オーディエンス（刺さる順）

1. **デザインできない PdM**（Kenji 型）— Figma で固まる。spec から動くものを自分で出したい。**最大の楔**。
2. **AI に前のめりなプロダクトデザイナー / デザインエンジニア**（Superset/Cursor を触り始めた層）— orchestration を理解、でもエンジニア向け UI に疲れている。
3. **エージェンシー / 受託デザイナー**（Tom 型）— 多ブランド量産・throughput・export。
4. **DS リード / エンプラ**（Priya 型）— 既存 DS 尊重・監査・統制。**初期は説得対象、本命は後**。

未来像：**ほとんどのプロダクトデザイナーが常駐する「もう一つのメイン」**。ファンが生まれ、語られる道具。

---

## 6. ブランドボイス

- **明晰 > 賢ぶり**。一文で言えることを三文にしない。
- **動詞で語る**（置く・描く・進める・握る）。名詞の羅列にしない。
- **あなた（you / デザイナー）を主語に**。「AI が〜」より「あなたが〜、エージェントが代わりに〜」。
- **誇張しない**。「革命」「世界を変える」は言わない。代わりに *体験で* 驚かせる（LP のデモ）。
- 日本語が主、英語のブランド語（Bezier / handle / curve）は原語のまま。

声のサンプル：
- ○「やりたいことを書く。ハンドルを置く。あとはエージェントが描く。」
- ○「あなたは判断する人。タイピストではなく。」
- ✕「AI が全自動であなたの開発を革命的に加速！」

## 7. ビジュアル原則（→ PRINCIPLES.md / design-tokens で詳細）

1. **作業面は静かに、生成物は主役に。**（継承）
2. **少数の制御点。**— UI 要素・色・操作を増やさない。ベジェのように、少ない点で全体を決める。
3. **連続体を可視化する。**（継承）intent→曲線（実装）が地続きに見える。
4. **手に馴染む、を品質バーに。**— Figma/Linear と並べて恥ずかしくない craft。ピクセル芝居でなく本物の心地よさ。
5. **融けて自然。**— 奇抜を避ける。気づかないほど自然な体験が最上。**モノクロを保ち、色相を足さない**（DEC-048。階層は濃淡で作る。機能色 destructive のみ例外）。

> **2026-06-12 改訂（DEC-048）**：当初は「handle-indigo を唯一のアクセント」とする方針だったが、ロゴをモノクロ "lit black"（抽象ペンツール）に確定したのに合わせ、ブランド全体を**完全モノクロ**に変更。indigo は全廃。詳細は `design-tokens.md` / `logo/README.md`。

## 8. ネーミング運用

- 正式表記：**Bezier**（英字・大文字 B・アクセント記号なし＝タイピング/URL/検索のため。`Bézier` は文中の語源説明でのみ）。
- 読み：ベジェ。
- プロダクト名・リポ名・パッケージ・ドメイン：すべて小文字 `bezier` を基本（識別子）。表示名は `Bezier`。
- タグライン（第一）：**Hold the handles.** / 「ハンドルを握る。曲線はエージェントが描く。」
- 旧名 `continuum` は痕跡を残さず置換（コード・docs・設定・repo 名・README）。

### 8.1 GTM 上の名前リスク と 運用ルール（ペルソナ/IDEO レビューで全員が指摘）
"bezier" は一般語（CSS `cubic-bezier`・各種ベクターライブラリ・ペンツール解説）と **強く衝突**する。検索第一想起・パッケージ名・ストア名の獲得は難しい。これは既知の負債として先に手を打つ:

1. **常に descriptor 付きロックアップで出す** — bare word 単体を避け、`Bezier — agent workbench`（または「— エージェント・ワークベンチ」）を基本表記に。第一印象で「曲線の道具」と誤読されるのを毎回打ち消す。
2. **発音キューを添える** — 「Bezier（ベジェ）」。口頭・社内展開（Kenji 指摘「読み方が分からん名前は広めにくい」）の摩擦を下げる。LP フッターと About に明記済み。
3. **資産を今すぐ確保** — ドメイン（`.app`/`.dev` 等、`bezier.com` は困難前提で代替を）/ npm・crates / GitHub org / 各 SNS ハンドル。識別子は `bezier` 取得困難なら `bezier-app` / `usebezier` 等の前置詞型を検討。
4. **ヒーローで即・カテゴリ明示** — 名前の誤読は「最初の画面でこれは agent workbench だ」と一瞬で示す実装（LP ヒーローの具体デモ）で打ち消す。calm より clear を最初の出会いだけ優先（IDEO）。

> これらは「名前を変える」理由にはしない（thesis 内包の強さが衝突コストを上回る、と判断）。ただし放置すると criterion 5（語られるブランド）が静かに頭打ちになるため、GTM 着手時の必須チェックリストとする。
