# Bezier — デザイン原則

> Bezier の UI・LP・ロゴ・モーションが従う実装規範。`2026-06-12_brand-strategy.md` の下流、
> `2026-06-12_design-tokens.md`（具体トークン）の上流（＝判断基準）。
> 迷ったら **「Figma/Linear と並べて、デザイナーの手に馴染むか？」** で決める。

---

## 1. 5つの原則

### 原則1 — 作業面は静かに、生成物は主役に
UI 自身は主張しない。ユーザーの制作物（spec・プレビュー・diff）が画面の主役。
- 背景はニュートラル、装飾は最小。クロームは紙背に沈む。
- 色は情報のためだけに使う（状態・AI・危険）。「飾りの色」を置かない。
- ✅ 制作物が画面の 80%+ を占める / ✕ ツールバーやパネルが主張する

### 原則2 — 少数の制御点（Few control points）
ベジェ曲線は少ない点で全体を決める。UI も同じ。**要素・選択肢・操作を増やさない。**
- 1画面1目的。タブ・モード・設定を増やす前に、消せないかを問う。
- 機能を足すコストはゼロに見えて、実は「制御点を増やす」コスト。楔から逸れたら足さない。
- ✅ 主要操作が片手で見渡せる / ✕ 「これも AI で作れるから」で機能が増殖

### 原則3 — 連続体を可視化する（Continuity）
intent → 曲線（実装）が分断されず地続きに見える。タブ往復で「別アプリ感」を出さない。
- spec・プレビュー・エージェントの作業は **同じ視野**に共存。切替でなく並置。
- 状態遷移は瞬間移動でなく、滑らかに（曲線のように）繋がって見せる（§4 モーション）。
- ✅ 1つの流れに見える / ✕ 画面が"別アプリ"に切り替わる

### 原則4 — 黒い画面を溶かす（Dissolve the terminal）
エージェントの実行は **隠さず、しかし脅さない**。
- 作業ログは黒コンソール調にしない。背景は作業面と地続き、文字は穏やかな階調、進行は「流れ」として読める。
- ユーザーはコマンドを打たない。**制御点（注釈・intent）を置く**だけ。
- 透明性は保つ（信頼の源）。「見えるが、要求されない」。上級者には本物の端末も渡す（奪わない）。
- ✅ ログが作業面に溶けている / ✕ ハッカー風の緑文字・黒背景で威圧する

### 原則5 — 融けて自然（Calm / "融けるデザイン"）
奇抜を避ける。気づかないほど自然な体験が最上。
- トレンド装飾（グラデ過多・ガラス・ネオン・過剰アニメ）を入れない。
- **モノクロ。** ブランドに色相を持たない。階層は色でなく**濃淡**で作る（DEC-048）。
- 「おっ」と思わせるのは LP のデモ体験で。アプリ内は静かに気持ちよく。
- ✅ 30分使って疲れない / ✕ スクショ映えのための装飾

## 2. レイアウト & スペーシング

- グリッド：4px / 8px。余白は **多め**。詰め込まない。
- 角丸：統一スケール（`--radius` 由来）。鋭角・極端な丸の混在を避ける。
- 階層は **余白と階調** で作る。線（border）を多用しない。線は最後の手段。
- コンテンツ幅：本文は読みやすい行長（~70 字）。LP の hero は中央寄せで呼吸を持たせる。

## 3. タイポグラフィ

- 見出し：やや重め・トラッキング詰め気味・行間ゆったり。
- 本文：可読性最優先。純黒でなく柔らかい前景色。
- 数値・コード・パス・セレクタ：**モノスペース**（ベジェの「数値」側を体現。情報の質感を変える）。
- 日本語と英数字が混じる前提で組む（フォントスタックは system stack、§tokens）。

## 4. モーション & インタラクション（"曲線のように"）

> モーションは Bezier の名前を体現する数少ない場所。**派手にせず、しかし"質感"を出す。**

- **イージングは必ずベジェ。** 線形（linear）を使わない。標準は緩やかな ease-out / ease-in-out。
  - 標準カーブ：`cubic-bezier(0.22, 1, 0.36, 1)`（"handle" — 出だし速く、終わり静か）。
  - 入場：`cubic-bezier(0.16, 1, 0.3, 1)`。
- **速い。** トランジションは 120–220ms を基本。"待たされ感"を作らない（Linear 基準）。
- **可逆・非破壊。** AI の提案は inline・取り消し可能・拒否可能。不可逆に適用しない。
- **状態は溶けて変わる。** 出現/消滅は opacity + わずかな translate/scale。飛び跳ねない（bounce 禁止）。
- **フォーカス遷移は連続的に。** パネル切替で文脈を失わせない。
- ホバー/プレスは即時の微フィードバック（背景の微変化）。クリックできることが触る前に分かる。

## 5. カラーの使い方（詳細は design-tokens）

- **モノクロ（DEC-048）。** ink + グレーのみ。ブランドに色相を持たない。差し色のアクセントを置かない。
- 階層は **濃淡**で作る。プライマリ = ink（light）/ near-white（dark）。
- ニュートラル基調（light / dark 両対応）。やや温かみのある off-white と true dark。
- **唯一の非ニュートラル = `--destructive`（機能的な赤）。** それ以外に色相を出さない。状態色（成功/警告/危険）は機能であり、控えめな彩度で。通常 UI はモノクロで成立させる。
- 純白(#fff)・純黒(#000)を大面積に使わない。

## 6. ロゴの扱い（詳細は logo/README）

- マーク = **抽象ペンツール**：四角アンカー（リポの起点）＋ハンドル線＋丸つまみ（握る制御点）＋曲線（エージェントが描く）。中空ノード＋ collinear ハンドル。
- 色は **モノクロ "lit black"**（光の当たりで微 sheen グラデ＝ベタ塗りでない）。`currentColor` でテーマ追従。色のアクセントは持たない。
- 四角と丸は**必ずハンドル線で結ぶ**（繋がってこそベジェ）。境界は均一な gap で確保する。
- 最小サイズ（≤24px）でノード中空が潰れる場面は **塗りつぶし版**（`mark-favicon`）に切替。
- 余白（クリアスペース）はマークの高さ分を四周に確保。

## 7. 禁則（やってはいけない）

- [ ] 純白・純黒を大面積に使う
- [ ] 色相を出す（モノクロ違反。許されるのは機能色 destructive のみ）
- [ ] エージェントログを黒コンソール調・緑文字で威圧する
- [ ] タブ/モードを増やして「別アプリ化」する（連続体を壊す）
- [ ] AI の提案を不可逆に適用する（必ず採否可能に）
- [ ] linear イージング・bounce アニメ・過剰モーション
- [ ] トレンド装飾（グラデ過多・ガラス・ネオン）で"映え"を狙う
- [ ] 制御点（UI 要素）をコスト無自覚に増やす

## 8. レビュー時の自問（実装前チェックリスト）

- [ ] これは Figma/Linear と並べて手に馴染むか？
- [ ] 制御点（要素・選択肢）を増やしていないか？1つ消せないか？
- [ ] モノクロを保てているか？（色相を足していないか。機能色 destructive 以外）
- [ ] エージェントの作業は「溶けて」見えるか？威圧していないか？
- [ ] イージングはベジェか？速いか（≤220ms）？
- [ ] ユーザーは常にハンドルを握っているか（可逆・拒否可能）？
