## 2. v0.1 runtime audit — 1 bug found & fixed

The build chain was green on v0.1, but a runtime audit found one real defect that the chain structurally cannot catch.

**BUG (fixed): frontmatter `created` date corrupted to a full ISO timestamp**

- File: `app/src/lib/markdown.ts`, `toFrontmatter()`.
- The YAML reader parses a date-only scalar (`created: 2026-06-08`) into a JS `Date` (midnight UTC). The code did `data.created.toISOString()`, producing `"2026-06-08T00:00:00.000Z"` — so opening then editing frontmatter silently rewrote a clean date into a noisy timestamp.
- Why the chain can't catch it: the value is type-correct (still a `string`), and the round-trip test only asserts on `rawFrontmatter` byte-preservation — it never serializes the parsed `data` object, so the corruption path is never exercised.

**Fix**: added `dateToYamlString(d: Date)` — returns `YYYY-MM-DD` when the `Date` is exactly midnight UTC (a date-only scalar), and only falls back to full ISO when an actual time component is present. `toFrontmatter()` now routes `Date` values through it. Re-serialization is now identical to source for date-only `created` fields.

## 3. What v0.2 adds

Two features, both layered on top of the v0.1 editor without changing its contracts:

### A. Embedded terminal (xterm ↔ Rust portable-pty)
A real pty-backed terminal pane inside the workspace window.

- **Backend**: `portable-pty` (wezterm) spawns a child in the workspace root; stdout/stderr are streamed to the frontend over Tauri events. The Rust side owns a `PtyState` session map keyed by uuid (v4).
- **Event contract** (frozen): Rust emits `pty://data` `{ id, chunk }` (UTF-8 lossy output) and `pty://exit` `{ id, code }`. Commands: `pty_spawn`, `pty_write`, `pty_resize`, `pty_kill`, `command_exists`.
- **Frontend**: xterm.js (`@xterm/xterm` + `addon-fit` + `addon-web-links`), dynamically imported with `ssr:false`. One active terminal at a time; the pane is hidden via CSS (not unmounted) so the pty session survives panel collapse. Changing the workspace root resets the terminal (old session pointed at the old folder).

### B. Doc-context → CLI agent handoff
"Hand off current doc" launches a locally-installed CLI coding agent seeded with the open document(s) as context.

- **Detection**: `detectAgents()` probes known agents (`claude` = Claude Code, `codex` = Codex) via `command_exists` on PATH; the picker shows only available ones.
- **Handoff**: `buildHandoff(root, docPaths, stamp)` reads each open doc and writes a single markdown context file to `<root>/.continuum/handoff/<stamp>.md`, ending with an instruction line telling the agent to read that file and implement.
- **Launch**: `launchSpecForAgent(tool, handoffPath)` produces a `{ cmd, args, initialInput }` spec. The workspace spawns a **new** terminal running the agent in the workspace root, then writes `initialInput` ("Please read <path> and implement.") into the pty once the session is ready.

## 6. KNOWN LIMIT — human gate

The 5-step build chain verifies **compilation and static correctness only** (TypeScript, ESLint, corpus round-trip, Next.js static export, Rust backend). It does **not** exercise the terminal or agent runtime:

- whether `pty_spawn` actually attaches a working shell,
- whether xterm renders/streams output live and resizes correctly,
- whether `detectAgents` finds the real binaries, the handoff file is written, and the agent actually reads it and acts.

These paths require running `npm run tauri dev` and driving the UI by hand. **v0.2 is build-green but runtime-unverified — a human gate is required before trusting the terminal/agent features.**

Also note: `buildHandoff` writes into `<root>/.continuum/handoff/`, and the Rust `write_file` command canonicalizes the parent dir — so that directory must already exist. A dedicated ensure-dir command is intentionally out of scope for the v0.2 frozen contract; a fresh workspace may need the directory created first.

## 7. v0.2 OUT (deliberately not in scope)

- Multiple concurrent terminals / multiple concurrent agents (one active terminal at a time).
- ensure-dir / mkdir-p Rust command for the handoff directory.
- Persisting terminal scrollback or agent sessions across app restarts.
- Streaming the agent's structured output back into the editor (handoff is one-way: doc → agent).
- Any agent beyond `claude` / `codex`; no remote/cloud agents.
- Runtime/integration tests for pty + agent paths (left to the human gate).

## 8. Suggested v0.3

Pick one of two directions:

- **Multi-agent / git worktrees** — allow several agents to run in parallel, each in its own git worktree off the workspace root, so concurrent handoffs don't collide. Adds a session manager UI (list/focus/kill) and a worktree lifecycle in Rust. Natural extension of the now-frozen pty + handoff contracts.
- **Canvas** — a spatial view over the workspace docs (and their frontmatter graph), making the doc → agent relationships visible and letting you hand off from a node directly. Leans on the v0.1 editor + v0.2 handoff rather than the terminal layer.

Recommendation: do the runtime human-gate pass on v0.2 first; let what breaks there inform which v0.3 direction to take.
