## 2. What v0.3 adds — the Canvas

v0.3 introduces a **Canvas board**: a spatial, pan/zoom view of the prototype as a set of live screens, laid out on an infinite board. It sits alongside the v0.1 editor + v0.2 terminal/handoff without changing their contracts.

### A. Canvas board of screen iframes (react-flow / @xyflow/react)
- A pan/zoom/drag board (`@xyflow/react` ^12.11.0) where each node is a **live screen frame**. Background grid, zoom Controls, and a pannable/zoomable MiniMap.
- Client-only: loaded via `next/dynamic({ ssr:false })` from the workspace route so `next build` (`output:"export"`) does not try to SSG react-flow.
- Node positions are owned by the parent (not react-flow internal state): on drag-stop the board calls `onMove(id, x, y)` and the parent persists; between drags the board re-derives nodes from the parent's `screens` so `screens.json` stays the source of truth.

### B. Gallery (add / pick screens)
- A gallery/list view of all screens in the workspace, with add + select affordances, complementing the spatial board.

### C. Add / move / remove screens — persisted to `.bezier/screens.json`
- A frozen Canvas **SoR** (`app/src/lib/screens.ts`): `Screen { id, label, source, x, y, w, h }`, `ScreensDoc { schema:"1", screens[] }`, persisted to `<root>/.bezier/screens.json`.
- `loadScreens` (missing/unreadable → empty doc), `saveScreens` (pretty-printed, stable key order for clean Git round-trips), `newScreenId` (stable `screen-N`, **not** wall-clock based).
- Add via the "Add screen" UI; move via drag (persisted on drag-stop); remove via per-node × button or Backspace/Delete.

### D. Screen sources: url / html / scenegraph
`ScreenSource` is a tagged union; `ScreenFrame` renders each:
- `url` → `<iframe src={url}>`
- `html` → `readFile(path)` → `<iframe srcDoc={html}>`
- `scenegraph` → `readFile(path)` → `JSON.parse` → a structured preview card (label + node list + reused badges). **Not** a pixel render (deferred to v0.4).

### E. Interact toggle
- `interactive=false` (default): a transparent overlay sits over each frame so nodes are selectable/draggable on the canvas but **not** clickable-through.
- `interactive=true`: the iframe receives pointer events, so you can actually **touch the prototype** inside the frame.

### F. SoR dir auto-create (Rust)
- `write_file` in `src-tauri/src/lib.rs` now `create_dir_all`s the parent tree if missing (e.g. `<root>/.bezier` on first save), after the existing `..`-traversal guard and before canonicalize. This closes the v0.2 known-limit where `.bezier/` had to pre-exist.

## 5. KNOWN LIMIT — human gate

The 5-step build chain verifies **compilation and static correctness only** (TypeScript, ESLint, corpus round-trip, Next.js static export, Rust backend). It does **not** exercise the Canvas at runtime:

- whether the react-flow board actually mounts in the Tauri webview and pans/zooms/drags,
- whether url/html iframes load and the scenegraph preview renders,
- whether the Interact toggle correctly passes pointer events through,
- whether add/move/remove actually persist to and reload from `.bezier/screens.json`.

These paths require running `npm run tauri dev` and driving the UI by hand. **v0.3 is build-green but canvas/iframe runtime is unverified — a human gate is required before trusting the Canvas features.**

Additionally:
- **Element editing is deferred to v0.4** (planned: Onlook). v0.3 frames are view/interact only — you cannot edit elements inside a screen.
- **The spike repo → scene-graph pipeline is not yet wired.** `scenegraph` sources render as a structured preview, but nothing yet extracts a scene graph from a repo or generates one back — that extract/generate path remains unwired.

## 6. v0.3 OUT (deliberately not in scope)

- In-frame element editing / mutation (→ v0.4, Onlook).
- Pixel-accurate scenegraph rendering (v0.3 is a structured preview card only).
- Wiring the spike's repo → scene-graph extract/generate pipeline.
- Edges/links between screens (frames are independent nodes; the SoR has no edge concept yet).
- Runtime/integration tests for the canvas + iframe + persistence paths (left to the human gate).

## 7. Suggested v0.4

- **Onlook element editing** — bring in Onlook so screens become editable surfaces (React + Tailwind): select an element in a frame and edit it, with changes written back to source. This is the natural next layer on the now-frozen Canvas SoR + ScreenFrame contracts.
- **Wire the extract/generate spike** — connect the repo → scene-graph pipeline so `scenegraph` sources are produced from real repos and can be generated back, turning the v0.3 preview card into a true round-trip source.

Recommendation: do the runtime human-gate pass on v0.3 first; let what the Canvas does (or breaks) at runtime inform how the Onlook editing layer and the extract/generate wiring are sequenced.
