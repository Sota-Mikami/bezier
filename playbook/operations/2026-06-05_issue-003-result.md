### 技術的ハマりと解決

1. **`_` 始まりディレクトリ問題**: Next.js App Router は `_` 始まりフォルダを private folder として route から除外する。`_continuum_preview` → `continuum-preview` に改名で解決。

2. **AuthGate ブロック**: root `app/layout.tsx` が AuthGate を含み、preview ルートも Supabase auth を要求した。AuthGate の先頭に `usePathname` チェックを追加し、`/continuum-preview` パスを bypass する最小変更で対処。hooks ルール（条件後に useState 不可）のため `AuthGateInner` 関数コンポーネントに分割。

3. **ReviewSession の Supabase 依存**: `fetchDueItems()` を `useEffect` 内で呼ぶため、preview では「loading→empty」で止まる。これはコンポーネントの設計上の制約であり、フォールバック（ラベル付きプレースホルダ）として明示した。

## 次の論点（ISSUE-004 候補）

1. **ReviewSession の data layer 問題**: 実部品をフル動作させるには、Supabase 依存の `fetchDueItems()` / `gradeSrsItem()` を preview 環境でモックする仕組みが要る。選択肢:
   - 型互換のモック関数を env 変数で差し替える（clean だが実装コスト中）
   - ReviewSession を fork した PreviewReviewSession を生成する（使い捨て OK なら）
   - そもそも ReviewSession の全機能は不要 → scene-graph の "この部品を使う" という情報だけで十分 → 割り切ってフォールバックのままにする

2. **preview ルートの汎用化**: 現在は chom-chom 固定のハードコード。`gen-*.json` を読んで動的にノードを組み立てる仕組みに昇格させると、任意 repo の preview が自動生成できる（Launch スコープ）。

3. **continuum app への統合**: preview ルートを `app/` 本体の canvas に統合する。ここで初めて "リアルタイムで scene-graph を編集してリレンダリング" という UX ループが生まれる。

4. **AuthGate の本格解決**: 現在の bypass は最小変更だが、長期的には continuum 専用の "sandbox mode" フラグ（env var）で制御する方が安全。
