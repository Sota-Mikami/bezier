## 採用経路: 経路A（Agent SDK + MCP stdio）

### 検証した経路

**経路A**: `@anthropic-ai/claude-agent-sdk` の `query()` に `mcpServers` でコンポーネントインデックスを MCP stdio サーバーとして渡す。

**経路B（フォールバック）**: `claude -p --mcp-config` で子プロセスを駆動する方法も事前検討したが、経路Aが動作したため採用しなかった。

### 経路Aの実装概要

| ファイル | 役割 |
|---|---|
| `spike/mcp-catalog.mjs` | MCP stdio サーバー。`search_components` / `get_component` / `get_tokens` / `emit_screen` の 4 ツールを公開。`EMIT_OUT_PATH` 環境変数で scene-graph を一時ファイルに書き出す |
| `spike/generate-sdk.mjs` | Agent SDK `query()` 呼び出し。`mcpServers` に mcp-catalog を stdio サーバーとして登録。`allowedTools` で MCP ツールのみ許可 |
| `spike/cli.mjs` | `Bezier extract` / `Bezier generate` / `Bezier list-indexes` の CLI エントリポイント |
| `spike/scene-graph-schema-v1.json` | scene-graph の JSON Schema（ローカル ↔ クラウド SoR コントラクト最小版） |

### 認証

`@anthropic-ai/claude-agent-sdk` は内部的に Claude Code の実行環境（`claude` CLI）を起動する。CEO の Claude Code サブスク認証がそのまま使われる。**ANTHROPIC_API_KEY 不要**。

## scene-graph schema v1（最小）

`spike/scene-graph-schema-v1.json` に JSON Schema として定義。

### フィールド

| フィールド | 必須 | 説明 |
|---|---|---|
| `schema_version` | ✅ | `"1"` 固定。クラウド SoR での互換性管理 |
| `label` | ✅ | 画面の人間可読なラベル |
| `route` | - | ルートパス |
| `source_repo` | - | 抽出元 repo パス（再現性） |
| `generated_at` | - | 生成日時（ISO 8601） |
| `intent` | - | 生成インテント文字列 |
| `nodes` | ✅ | scene-graph ノードのリスト |
| `reused` | ✅ | 流用した既存パーツ名のリスト |

### node_kind

- `existing_component`: 既存パーツ流用（`component` フィールドで名前を指定）
- `generated`: 新規生成（最小限）
- `primitive`: HTML 相当の原始要素

### バージョン管理方針

- `schema_version` は文字列。次の破壊的変更時に `"2"` にインクリメント
- クラウド SoR は schema_version ごとにマイグレーション処理を持つ（次 ISSUE）

## 観測した課題・次の論点

### 論点1: MCP ツール実行権限

alloy の生成で Agent SDK がデフォルトの built-in ツール（LSP）を呼ぼうとした。`allowedTools` で制限をかけたが、「MCP ツールのみ許可」するには `tools: []` + `allowedTools` の組み合わせが必要。本体実装では permission 設計を整理する。

### 論点2: コスト（$0.59〜$1.80 / 生成1回）

現状 Agent SDK が Claude Opus 4.8 [1m] を使っている（CEO のサブスクのデフォルト）。catalog が大きい repo（chomchom=67 parts）はコストが高い。対策:
- `model: 'claude-sonnet-4-6'` を options に指定（未検証。次 ISSUE で確認）
- catalog の prompt caching（generate.mjs の既存実装を流用）

### 論点3: emit_screen の受け渡し方式

現状は `EMIT_OUT_PATH` 環境変数 → 一時ファイル → 親プロセス読み込み という迂回路。Agent SDK に「MCP ツール呼び出し結果をストリームから取得する」方法があるか確認する。今は動くが本体ではよりクリーンな実装に変えたい。

### 論点4: scene-graph → render（L3）

今回は scene-graph 生成まで。次は scene-graph から Playwright render で実スクリーンショットを出し、「本当にプロダクトらしいか」を side-by-side で確認する（ISSUE-001 残タスク）。`capture-screens.ts` の流用。
