## 2. What v0.4 adds

v0.4 introduces the **scaffolding** for Onlook-style element editing of `react-repo` screens on the Canvas: click an element in a live preview, edit its Tailwind classes in an inspector, see an instant preview, and (when the repo is instrumented) write the change back to source. It sits alongside v0.1–v0.3 without changing their contracts.

### A. Vendored Onlook parser subset (Apache-2.0) — `app/src/vendor/onlook/`
A trimmed, dependency-localized copy of Onlook's AST write-back engine:
- `parse.ts`, `ids.ts`, `helpers.ts`, `packages.ts` (Babel re-exports), `constants.ts`, `code-edit/{helpers,style,transform}.ts`, plus `index.ts` (the curated public surface).
- `@onlook/*` workspace imports were localized: `@onlook/constants` → inlined `constants.ts`; `@onlook/utility` `createOid`/`createDomId` re-implemented (nanoid `customAlphabet`); `customTwMerge` → plain `twMerge` (`tailwind-merge`); `@onlook/models` → a tiny local `CodeDiffRequest` type.
- Babel engine = `@babel/standalone` (same as upstream).
- **Scope reduction**: `transform.ts` handles **className + prop edits only**. Upstream's structural-edit branch (group/insert/move/remove/image/text) was dropped — out of scope for v0.4.

### B. `react-repo` screen source — `app/src/lib/screens.ts` + `add-screen.tsx`
New tagged-union member: `{ type: "react-repo", url, repoPath }` — `url` is the dev-server URL shown in the iframe; `repoPath` is the local repo root for write-back. The "Add screen" form gained a "React repo (editable)" option with the second `repoPath` field. This is the **only** editable source type.

### C. Editable preview + bridge — `editable-frame.tsx` + `public/bezier-preview-bridge.js` + `lib/preview-bridge.ts`
- `EditableFrame`: renders the dev URL in an `<iframe>`, speaks a hand-rolled `postMessage` protocol, exposes an imperative handle (`applyStylePreview`, `highlight`), shows a bridge-connected dot, and renders a **clear "not instrumented" overlay** if no `ready` handshake arrives within 4 s.
- `bezier-preview-bridge.js`: the **child** script that must run inside the target app's iframe — click→select, dblclick→open-source, lazy `data-odid` assignment, `getComputedStyle` subset, live class apply/replace, outline highlight.
- `preview-bridge.ts`: the shared, DOM-free protocol (`BRIDGE_NAMESPACE`, message types, `StyleEdit`, type guards) imported by both sides so they cannot drift.

### D. Element inspector — `element-inspector.tsx`
Controlled panel: editable Tailwind-class textarea, quick-class chips, read-only computed-style list, "replace all" override, Apply/Reset, and a write-back status line. Owns **no** file I/O.

### E. Write-back pipeline — `app/src/lib/onlook-edit.ts`
- **Pure/Node-safe**: `instrumentContent` (inject `data-oid` + collect `oid → {tag,line,col,component}`), `applyEditToContent` (Tailwind-aware className/prop merge via the vendored `transformAst`).
- **Tauri-only** (uses `ipc.readFile/writeFile`): `instrumentFile(s)`, `loadOidIndex`/`saveOidIndex`, `applyEdit`. The `oid → {file,...}` sidecar persists at `<repo>/.bezier/oid-index.json`.

### F. Canvas wiring — `canvas-board.tsx` + `workspace/page.tsx`
"Edit" toggle (Board mode, requires a `react-repo` screen) swaps `ScreenFrame` → `EditableFrame`; selection docks the inspector; Apply does live-preview then dynamically imports `onlook-edit` for source write-back; double-click resolves `oid → file` and opens it in the Editor. The heavy pipeline (vendored Babel) is `import()`-ed at call time so it stays out of the initial/SSG bundle.

## 3. The Onlook technique (scout findings, grounded in real source @ `a242be58`, Apache-2.0)

- Onlook injects an **opaque** `data-oid` (7-char nanoid) into JSX **at build time** and keeps a **separate persisted map** `oid → {file, line, col, component}`. **It does NOT put `file:line:col` in the DOM** — only the opaque key. A clicked node maps back to source by string-equal `oid`.
- A second attribute `data-odid` is a **runtime** DOM id, assigned lazily by the in-iframe preload script (addressing handle for editor↔iframe).
- Upstream uses **penpal** for the iframe bridge; Bezier **hand-rolls** the equivalent `postMessage` protocol to keep the dependency surface small — same message intent, smaller surface.

Bezier mirrors this exactly: opaque `data-oid` durable key, sidecar index at `.bezier/oid-index.json`, lazy `data-odid` in the bridge, computed-style subset, stylesheet-free live preview by mutating the node's class list.

## 4. WHAT ACTUALLY WORKS vs WHAT IS STUBBED / NOT WIRED

### ✅ Works (verified by compile/type/lint, and ad-hoc for the pure functions)
- **The vendored AST engine compiles and runs as pure functions.** `instrumentContent` injects one `data-oid` per JSX element and reports locations; `applyEditToContent` merges/replaces `className` Tailwind-aware and regenerates source with a minimal diff. These are Node-safe and were exercised ad-hoc end-to-end (instrument → transform → regenerate) during implementation.
- **All UI is wired and renders**: Edit toggle, `react-repo` add-screen flow, `EditableFrame` iframe, the postMessage bridge handshake/selection/live-preview/highlight/open-source paths, the inspector, and the not-instrumented overlay. These are real React wiring, not placeholders.
- **Live visual preview works without instrumentation**: clicking an element and hitting Apply mutates the live DOM class list in the iframe even if the element has no `data-oid` (the inspector says so explicitly).

### ⚠️ Stubbed, not wired, or unproven (the honest part)
1. **Instrumentation is never invoked from the product.** `instrumentFile(s)` exist in `lib/onlook-edit.ts` but **nothing in the app, no script, and no CLI calls them.** There is no button, no build step, no babel plugin. Consequently **no `oid-index.json` sidecar is ever produced through Bezier.**
2. **Therefore end-to-end source write-back is NOT demonstrably functional yet.** `applyEdit` resolves `oid → file` via the sidecar; with no sidecar it throws `OidNotFoundError`. A user cannot today: open Bezier → add a react-repo screen → click → edit → see source change, because the instrument step has no entry point. Write-back works *in principle* (the functions are correct in isolation) but the **product loop is broken at the instrument stage.**
3. **No babel/build-time instrumentation plugin shipped.** Onlook injects `data-oid` at build time via a babel plugin (scout). Bezier's approach is a one-shot source rewrite (`instrumentFiles` writes `data-oid` into the source files), which is a different, simpler model — but it has no trigger and was never run against a real repo in-app.
4. **Never run against a real running React+Tailwind app inside Tauri.** No integration test, no manual proof this session beyond pure-function checks. The full chain (running dev server → bridge connects → click → edit → file write → HMR reflects it) is **unverified**.
5. **Structural edits unsupported.** className + prop only. No insert/move/remove/group/text/image (intentionally out of scope).
6. **Live-preview vs source-merge can diverge.** The DOM preview *appends* classes; the source write uses `tailwind-merge`. For conflicting utilities the previewed pixels may differ slightly from the post-save render until the iframe reloads.
7. **`roundtrip.mjs` does not cover any v0.4 code** (markdown only). The v0.4 engine has **no committed regression test.**

## 5. Setup a target React+Tailwind app needs to become editable

For a `react-repo` screen to be *selectable + live-previewable*:
1. **Run its dev server** and point the screen's `url` at it (e.g. `http://localhost:3000`).
2. **Load the bridge dev-only**: copy `public/bezier-preview-bridge.js` into the target app and load it once in dev, e.g. in the Next.js root layout:
   ```tsx
   {process.env.NODE_ENV !== "production" && <script src="/bezier-preview-bridge.js" />}
   ```
   The iframe uses `sandbox="allow-scripts allow-same-origin allow-forms"`; `allow-same-origin` is required so the in-iframe bridge can call `getComputedStyle` and set `data-odid`.

For *source write-back* (in addition to the above):
3. **Instrument the source with `data-oid`** so DOM nodes map back to JSX, and produce the `<repo>/.bezier/oid-index.json` sidecar. Onlook does this with a **build-time babel plugin** (scout); Bezier currently exposes only the library functions (`instrumentFiles`) to do it as a one-shot source rewrite — **but provides no wired entry point to run them** (see §4.1). Until that exists, this step has to be done by hand / out-of-band, which makes write-back effectively unavailable to a normal user today.

## 7. Honest next steps to reach a working live editor (the human-iteration work)

In rough priority order:

1. **Wire instrumentation to an entry point.** Add a "Make editable" action (or a CLI / one-shot on add-screen) that walks the repo's `.tsx/.jsx`, runs `instrumentFiles`, writes the sidecar, and reports counts. Without this, nothing downstream is usable. Decide build-time babel plugin (Onlook model) vs one-shot source rewrite (current lib) — the latter mutates the user's source and needs a Git-clean guard + undo story.
2. **Prove the full loop against one real app.** Stand up a throwaway Next.js + Tailwind app, load the bridge, instrument it, and manually verify: click → inspector populates → Apply → file on disk changes → HMR reflects it → no spurious diff. This is the single most important missing validation.
3. **Add a committed test for the pure engine.** A small harness (mirroring `roundtrip.mjs`) that asserts `instrumentContent` idempotency and `applyEditToContent` className-merge correctness on fixtures, wired into the verification chain.
4. **Reconcile preview vs source merge.** Make the live DOM preview use the same `tailwind-merge` resolution as the source write so previewed pixels match post-save, or auto-reload the iframe after write-back.
5. **Cross-origin / dev-server ergonomics.** Document/handle the same-origin constraints; consider a small dev middleware or a documented snippet so users don't hand-edit their layout.
6. **Then, only if needed:** structural edits (insert/move/remove/text), and component-instance awareness (the `component` field in the sidecar is collected but unused).

**Bottom line**: v0.4 is an honest *skeleton* — a correctly-vendored, correctly-licensed Onlook engine plus full UI wiring and a working live-preview path. The element-editing **product loop is not yet closed** because instrumentation has no trigger and the whole thing has never been run end-to-end against a real app. The green verification chain means "doesn't regress, compiles clean," not "the editor works."
