## 3. M4 zero-diff behavior

The M4 **zero-diff clean-save** guarantee is intact:

- When a document is **not dirty**, the **original bytes are written back verbatim** on save — no reformatting, no whitespace churn, no reordering. A save that the user did not edit produces a byte-identical file.
- The **frontmatter raw block is preserved verbatim** through the round-trip.
- Behavior lives in `plate-editor.tsx` and `qa-table.tsx` and was confirmed preserved across all the lint refactors above (the remount/closure refactors changed *how* state is reset, not *what* bytes get serialized).

## 5. Known limitations

- **Round-trip runner is fragile under bare `node`** — depends on a deprecated `tsx --loader`; works only via the `npx tsx` fallback. The `node || npx tsx` chain masks this but it should be cleaned up.
- **Plate is dynamic / `ssr:false`** — the editor is client-only; no SSR of editing surface. Fine for the desktop (Tauri) target but means the static web build ships an empty editor shell until hydration.
- **Static export constraints** — `/issues/[id]` uses `generateStaticParams` (6 paths prerendered). Any new dynamic route needs the same treatment or the static export breaks.
- **Re-baseline via remount** — frontmatter/QA panels now reset state by parent `key` remount rather than effects. Correct and simpler, but means any future in-place baseline update (without changing the key) must be designed explicitly.
- **Raw-fallback path is untested by corpus** — 0 files currently exercise it, so its real-world behavior on exotic markdown is unverified against live data.
- **No automated test for the zero-diff invariant** — M4 is verified via the round-trip corpus, not a dedicated assertion that flips dirty/clean and diffs bytes.
