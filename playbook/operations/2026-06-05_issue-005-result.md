## 採用方式

**ShimEngine（`spike/shim-engine.mjs`）** を独立モジュールとして新規実装し、`cli.mjs` の `preview` コマンドに統合した。

シムの責任範囲:
1. AuthGate.tsx へのバックアップ → bypass 注入 → 復元
2. `.gitignore` へのシムブロック追記 → 復元（逆算除去）
3. `Bezier-preview/` ディレクトリの生成 → 削除
4. マニフェストによるクラッシュ耐性

`generate-preview.mjs` からは gitignore 操作・AuthGate bypass 注入コードを完全削除し、シムが作成したディレクトリにファイルを書くだけの責務に縮小した。

## 安全機構の実装（DEC-003 必須制約）

### ① gitignore 必須・絶対にコミットしない

追記するシムブロックは `"ISSUE-005 shim"` マーカーで識別し、終了時に正確に除去する。`Bezier-preview/` ディレクトリ自体は gitignore されるため、内部ファイルがコミットされることはない。AuthGate.tsx の変更もバックアップから復元するため、最終的に git の tracked ファイルに差分は残らない。

### ② 終了/クラッシュ時に原子的に自動復元

シム適用後は即座にマニフェストをディスクに書く（対象 repo の外 `spike/out/shim-manifest-{name}.json`）。正常終了・SIGINT・SIGTERM・uncaughtException のすべてで `revert()` を呼ぶハンドラを登録。

```
apply() 呼び出し
  → マニフェスト初期化 + 保存（クラッシュ耐性: ここで止まっても次回起動時に検出）
  → AuthGate.tsx バックアップ → 注入
  → マニフェスト更新
  → .gitignore 追記 → マニフェスト更新
  → Bezier-preview/ 作成 → マニフェスト更新
```

各エントリを追加するたびにマニフェストを保存するため、どの段階でクラッシュしても「適用済み分だけ」を復元できる。

### ③ クラッシュ後の手動復元（再起動後も復元できる）

```bash
node cli.mjs shim-restore chomchom
# または
node shim-engine.mjs revert chomchom
```

次回 `apply()` 呼び出し時にも既存マニフェストを自動検出して復元してから再適用する。

### ④ opt-out: `--no-shim`

```bash
node cli.mjs preview chomchom --no-shim
```

tracked ファイルを一切変更しない厳格 read-only モード。

### ⑤ 最小範囲のみ

変更対象:
- `AuthGate.tsx` 1ファイル（layout が import している gate のみ）
- `.gitignore` 1行ブロック
- `Bezier-preview/` ディレクトリ（生成 → 削除）

## 次の論点

1. **`--no-shim` の Bezier-preview 残留**: `--no-shim` では gitignore 追記もしないため、untracked の `Bezier-preview/` が残る。`--no-shim` でも dir_create を ShimEngine（ただし gitignore なし）で管理するか、cli.mjs の finally で削除するかを決める

2. **ReviewSession の FALLBACK**: 残り 1 FALLBACK。`"service-dependent (Session)"` 分類だが、実際は `ReviewSession` が Supabase セッションに依存しているためで、mock Session オブジェクトを渡す heuristics を追加すれば REAL にできる可能性

3. **template の AuthGate**: template の scene-graph には AuthGate が existing_component として登場するが、layout にはない。scene-graph 内の blocking gate をシムが検出して inline で bypass する heuristic が必要かもしれない

4. **他の gate パターン**: `PasswordGate`・`CommentLayer` 等への拡張。現状は `blockingComponents` リストで静的に管理しているが、layout の render tree を解析して動的に検出する方向が正解

5. **app/ canvas 統合**: preview ルートを Bezier app の canvas に統合し、scene-graph 編集 → リアルタイムリレンダリングのループを実現（Launch Stage 候補）
