## 1. スコープ

### IN（v0.1 でやる）
- Tauri v2 殻 + 既存 `app/`（Next.js/React/Tailwind/shadcn）を内包
- **ワークスペース** = ローカルフォルダを選んで開く（フォルダピッカー）
- 左 = **ファイルツリー**（.md/.mdx/.yaml を表示）
- 中央 = **エディタ**:
  - `.md/.mdx` → **Plate**（Notion 風ブロック、見出し/箇条書き/表/コード/引用/チェックボックス + frontmatter 編集）
  - `.yaml` → **構造ビュー**（後述の QA は表UI、それ以外は素の YAML テキスト + バリデーション）
- **保存 = 実ファイル書き戻し**（Plate ドキュメント → Markdown シリアライズ → ディスク）。往復で内容が壊れないこと（round-trip 安全性）が v0.1 の合否ライン
- frontmatter（`---` ブロック）を構造フィールドとして編集（title/type/status/created など）

### OUT（v0.1 ではやらない）
- 埋め込みターミナル / エージェント委譲（v0.2）
- Canvas（表示・要素編集）（v0.3 / v0.4）
- クラウド SoR / 共同編集 / 課金 / 認証（後）
- Onlook 統合・Node サイドカー（v0.4 まで不要）
- リアルタイム collab、複数ウィンドウ

## 2. ワークスペースとファイルモデル（SoR）

continuum のワークスペース = **ただのフォルダ**。中身は人間が手編集でき Git diff が出るファイル。continuum はそのビュー＋エディタ。

```
<workspace>/
├─ .continuum/
│   └─ workspace.json        # continuum 設定（表示順・お気に入り等。SoR ではない補助メタ）
├─ specs/      *.mdx          # 仕様（Plate 編集）
├─ research/   *.mdx          # 調査・施策メモ（Plate 編集）
├─ decisions/  *.mdx          # 意思決定ログ（Plate 編集）
└─ qa/         *.yaml         # テスト項目（表UI 編集）
```

> 既存の自分の playbook（`*.md` 直置き）もそのまま開ける設計にする（フォルダ構造を強制しない）。`.continuum/` が無ければ初回に生成。

### 正本フォーマット原則（DEC-006 再掲）
- **正本は常に .mdx / .yaml（手編集可・Git diff 可）**。Plate のブロックも YAML 表UIも、その上の**ビュー**。
- アプリ独自の JSON 状態をディスクの正本にしない。

### .mdx ファイルの形（例: spec）
```mdx
---
title: ログイン画面の刷新
type: spec
status: draft        # draft | in-review | approved
created: 2026-06-08
links: [decisions/DEC-012.mdx]
---

## 背景
...（Plate ブロックで編集される本文）...

## 要件
- [ ] メール+パスワード
- [ ] SSO（Google）
```

### .yaml ファイルの形（例: qa）
```yaml
title: ログイン画面 QA
type: qa
cases:
  - id: QA-001
    given: 未ログイン状態
    when: 正しい資格情報で送信
    then: ダッシュボードに遷移
    status: todo      # todo | pass | fail
```

## 3. Plate ブロック設計

### v0.1（標準ブロックで往復を固める）
標準 markdown ブロックのみ。**MDX round-trip を最優先で安定させる**:
- paragraph / heading(1-3) / bulleted・numbered list / todo(checkbox) / blockquote / code block / table / divider / link / inline(bold/italic/code)
- frontmatter エディタ（本文と分離した上部パネル。title/type/status 等のフィールド）

### v0.1.x（continuum の差別化＝独自ブロックを"増やせる"基盤）
ここが「ただの markdown エディタ」と continuum を分ける所。**MDX の JSX コンポーネントとして独自ブロックを定義**し、`.mdx` に `<Requirement/>` 等として永続化:
- **Spec 用**: `<Requirement priority=.. />` / `<AcceptanceCriteria/>` / `<DecisionLink id=.. />` / `<ReusedComponent path=.. />`
- **QA 用**: `<TestCase id given when then status />`（YAML と相互変換可能に）
- 設計方針: **ブロック = Plate plugin + MDX シリアライザ登録**の1ペアで増やせる構造にしておく（「block component を増やしていける」要件への回答）。

> v0.1 は標準ブロックで round-trip を固め、独自ブロックは v0.1.x で1個ずつ足す。最初から独自ブロックを盛らない（往復が壊れるリスクを先に潰す）。

## 4. 技術構成（DEC-006 準拠・最小）

| レイヤ | v0.1 の実装 |
|---|---|
| 殻 | **Tauri v2**（`app/` を内包）。FS アクセス = Tauri の fs/dialog プラグイン（Rust 側、**Node サイドカー不要**） |
| UI | 既存 `app/`（Next.js/React/Tailwind/shadcn）。静的書き出し or Tauri dev で起動 |
| エディタ | **Plate（MIT）** + `@platejs/markdown`（MD/MDX シリアライズ） |
| MDX 解析 | frontmatter = `gray-matter` 相当 / 本文 = Plate の MD シリアライザ |
| YAML | `yaml`（js）でパース・整形、QA は表UI |
| ファイルI/O | Tauri command: `read_file` / `write_file` / `list_dir` / `pick_folder`（Rust 標準 + tauri-plugin-fs/dialog） |
| Git | v0.1 は**触らない**（ユーザーが普通に git add/commit。continuum は diff を生むだけ）。git status 表示は v0.2+ |

### Next.js × Tauri の注意
- Tauri は静的フロントを期待 → Next.js は `output: 'export'`（静的書き出し）か、dev は Tauri の devUrl に Next dev を指す。**SSR/Server Actions は使わない**（ローカルアプリなので不要）。
- ルーティングはクライアントサイドのみ。

## 5. アプリ構成の変更（既存 app/ から）

現状 `app/` は「Design Issues 一覧 + 会話駆動詳細 + ダミーデータ(data.ts)」。v0.1 では:
- **新しいエントリ = ワークスペース・エディタ画面**（フォルダツリー + Plate）。これを主役にする。
- 既存の会話駆動カスケード UI（issues/[id]）は **v0.2 以降の土台として温存**（消さない・ルートを分ける）。
- `data.ts` のダミー駆動は editor からは使わない（実ファイル駆動に）。

新規ファイル（想定）:
```
app/src/app/workspace/page.tsx           # ツリー + エディタのレイアウト
app/src/components/file-tree.tsx
app/src/components/editor/plate-editor.tsx
app/src/components/editor/frontmatter-panel.tsx
app/src/components/editor/qa-table.tsx
app/src/lib/workspace.ts                  # Tauri command ラッパ（read/write/list/pick）

src-tauri/                                # Tauri プロジェクト（新規）
```

## 6. マイルストーン & 受け入れ基準（v0.1 完了の定義）

1. **M1 殻**: Tauri v2 で既存 app/ が起動する（空ウィンドウでなく一覧が出る）
2. **M2 開く**: フォルダを選ぶ → ツリーに .md/.mdx/.yaml が出る → クリックで中身表示
3. **M3 編集・保存**: .mdx を Plate で編集 → 保存 → ディスクのファイルが更新される
4. **M4 round-trip 安全**: 既存の自分の playbook の .md を開いて**無編集で保存しても diff がほぼ出ない**（順序・記号の崩れが許容範囲）。← **v0.1 の本丸の合否ライン**
5. **M5 frontmatter**: title/type/status を構造フィールドとして編集できる
6. **M6 QA**: .yaml を表UIで編集 → 保存で YAML が整形されて書き戻る

**Done = M1〜M6 を満たし、CEO が自分の playbook を continuum で1日編集して耐える。**

## 7. リスク & 対策

| リスク | 深刻度 | 対策 |
|---|---|---|
| **Plate ↔ MDX の round-trip でフォーマットが崩れる**（既存 .md が壊れる） | 高 | M4 を独立の合否ラインに。最初は標準ブロックのみ。崩れる記法は「未対応として原文保持」フォールバックを用意。**自分の重要ファイルはコピーで実験**（原本を壊さない） |
| Next.js を Tauri に載せる際の SSR/export 摩擦 | 中 | `output: export` + クライアントルーティングに最初から固定 |
| Tauri の Rust ツールチェーン導入コスト | 低 | 一度入れれば済む。fs/dialog プラグインで v0.1 は完結 |
| スコープが Canvas/ターミナルに漏れる | 中 | v0.1 は editor のみ。OUT を厳守（DEC-006 のビルド順） |
| 独自ブロックを盛りすぎて往復が不安定化 | 中 | v0.1 は標準ブロックのみ、独自は v0.1.x で1個ずつ |
