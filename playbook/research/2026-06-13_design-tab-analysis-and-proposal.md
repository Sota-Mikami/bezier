# Design タブ（考える層）— 不全の分析と再設計提案

> **CEO FB**: 「Spec → repo → Build は回る。が、**Design のパターンを見て・考えて・相談する**という体験が今できていない。しっかり分析して提案して。」
> **位置づけ**: DEC-051 で作った中央 Design タブ（HTML 別案）の作り直し提案。差別化の核なので妥協しない。

---

## 1. 実機の証拠（推測でなく事実）

- lenz の 2 Issue を実機で dogfood。**Build + Verify は動作**（implement/verify handoff・spec.md・verify.md が生成済）。
- **Design は 1 案も生成されていない**：`.bezier/drafts/*/design/` フォルダ無し・variant handoff 無し。= 「別案を作る」が**一度も成果物を出していない**。「使いにくい」ではなく「**機能していない**」。

---

## 2. なぜ使えないのか（根本原因 5 つ）

### ① 成果物のミスマッチ（最大の技術的欠陥）
- 実 repo は **React + Tailwind**（utility class で見た目が決まる）。一方 variant は **「自己完結・インライン CSS のみ・JS/CDN 非依存・`sandbox=""`」** を強制。
- → エージェントが自然に書く Tailwind class は **sandbox 内で無効**（Tailwind ランタイム無し）→ **崩れた無スタイル HTML**。回避しようと CSS を手書きインライン化すると **遅い＆実 DS と別物**。
- **「実部品に忠実」という Bezier の楔（ISSUE-003/004/005）に正面から反する。** v0/Lovable の白紙生成を批判してきたのに、ここで自分が白紙の崩れ HTML を出している。

### ② 生成が会話を破壊する
- `handleGenerateVariant` は **`ptyKillKey` → fresh `claude` 起動**。= 今やっているチャットを毎回**殺して**新セッション。1 クリック 1 案。
- 「見て・考えて・**相談する**」の"相談"＝会話の連続性が前提なのに、生成のたびに会話が途切れる。

### ③ チャット（考える場）と board（見る場）が分断
- 左で会話・中央で別案、だが**両者が繋がっていない**。「**B の方向で、もっと密に**」のように**案を指して話す**手段が無い。`@A` 参照も無い。
- 結果、Design タブは"生成して眺めるだけ"で、**対話的に詰める**ことができない。

### ④ 発散→反応→収束の loop が無い
- 1 案ずつ手で追加するだけ。「**まず 3 方向出して**」（発散）→「**B を軸に 2 つ**」（収束）というラウンド構造が無い。比較・採用・ブレンドの導線も弱い。

### ⑤ 入口の順序が逆
- 生成は worktree（＝Build セッション）必須。だが「**作る前に方向を見比べる**」のが Design の役目。**Build を始めてからでないと Design が使えない**＝心理的にもう"作るモード"で、探索が起きない。

---

## 3. そもそも Design 層は何のためか（再定義）

- **発散**：spec を満たす **複数のビジュアル方向** を、安く速く並べて**見比べる**。
- **相談**：それらを見ながら**エージェントと対話**して、好き/嫌い/混ぜたいを言語化し、次の案へ。
- **収束**：1 案 or ブレンドを**採用**し、「**なぜこの方向か**」を残して **Build** へ渡す。
- 実 preview（Build）との棲み分け：Build＝**本物 1 つを作り込む**。Design＝**まだ作らずに方向を決める**。**考える層は使い捨て、決定だけが資産**（DEC-014/A と整合）。

→ 現状はこの 3 つ（発散・相談・収束）が**どれも成立していない**。

---

## 4. 提案：Design を「ビジュアルに接地した設計対話」にする

体験の芯は **board（見る）× chat（相談）の連結**。生成の中身（fidelity）は §5 で分岐するが、**以下の体験骨子は fidelity に関係なく正しい**。

### A. チャット接地（相談を成立させる）
- Design の生成・反応は**今の会話の続き**で行う（kill しない＝`--continue`）。会話を壊さない。
- **案を参照できる**：チャットに `@A` `@B`（または案カードの「これを基に話す」）。「**@B をもっと密に**」「**@A の余白＋@C の構成**」が言える。
- Design タブ自体に**軽いコンポーザ**（「3 方向出して」「@B を密に」）。送ると board が増える。＝board と chat が同じループ。

### B. バッチ＆ラウンド（発散→収束）
- 「**N 方向を一度に**」（既定 3）。1 ターンで複数案を生成し、各案を 1 行で説明。
- ラウンド：round1 発散 → 反応 → round2 は採用案を軸に refine＋ワイルドカード 1。**履歴（どの round で何を選んだか）** を残す。

### C. 参照パターンを入口に（"見て考える"の素材）
- 「**他社はこの UI をどうしてる？**」を Design の冒頭に。Mobbin MCP / インスピレーションの**参照パターン**を数枚見せ、「この vibe で」を選んでから applied 案へ。
- ＝**発散の質が上がる**（白紙からでなく、良い前例から方向を選ぶ）。任意ステップ。

### D. 収束を資産に
- **採用 / ブレンド**を明示アクション化 → 「**なぜこの方向か**（decision メモ）」を残し、`handlePickVariant` で Build に方向を渡す（既存）。HTML は throwaway、**decision が資産**。

### E. 入口の順序
- Design は **spec 確定後すぐ**（重い Build 前）に入れる。生成にエージェントは要るが、"探索セッション"として軽く始められるようにする（worktree は要るが、コードは触らず design/ だけ書く＝今の実装の意図に近い。ただし§2②③⑤を直す）。

## 5. 中核の分岐（CEO 判断ポイント）：別案の "fidelity"

「見て考える」素材を**どれだけ本物に寄せるか**。ここが体験と工数を最も左右する。

| 案 | 中身 | 長所 | 短所 |
|---|---|---|---|
| **W. 正直なワイヤー** | グレースケールの構造スケッチ（plain inline CSS・速い・sandbox OK） | 速い・安い・repo 非依存・**崩れない**・"使い捨て"が honest | 実 DS の見た目は分からない（構造/レイアウトの比較に限定） |
| **R. 実 DS で実描画** | repo の**実部品**で throwaway の "design ギャラリー"（route/component）を作り、**実 preview** で見る | **忠実＝Bezier の楔そのもの**・本物の見た目で判断 | 重い・repo 固有（routing 等）・dev server 必須・scaffolding が難 |
| **H. ハイブリッド（推奨）** | **まず W で発散**して方向を決め、**採用案だけ R で実描画**して確認 | 安く発散→忠実に収束。両取り | 2 段階で実装はやや増える |

- **Principal Designer + Head of Product 推奨 = H（ハイブリッド）**。理由：
  - 発散は**安く・速く・多く**が命（W が最適）。だが「構造スケッチだけ」では CEO の"見て考える"に物足りない懸念 → **採用候補は実 DS で実描画（R）**して納得して決める。
  - **楔（実部品忠実）を収束フェーズで効かせる**。白紙生成批判と矛盾しない。
  - 現状の「静的 HTML を実 DS に偽装」＝**W でも R でもない中途半端**だったのが①の崩れの原因。W は"スケッチと割り切る"、R は"本物"。どちらかに倒すのが正しい。

---

## 6. 段階導入（安価×高価値から）

1. **体験骨子（§4 A/B）＝最優先**：チャット接地（kill しない・`@案` 参照）＋バッチ生成（3 方向一括）。**fidelity 据え置きでも体験は激変**。＝②③④の解消。
2. **fidelity を W に倒す**：別案を「**グレースケール構造ワイヤー**」と明示（プロンプト改訂・sandbox 維持）。①の崩れが消え、発散が速く安くなる。
3. **R（実描画）を収束に**：採用候補を実 preview で見る経路。Build の preview infra を流用。←ハイブリッド完成。
4. **参照パターン（§4 C）**：Mobbin/インスピレーションを入口に。
5. **decision 記録（§4 D）**：採用理由を残す。

## 7. CEO への質問（提案を確定するため）

1. **fidelity の方向**：W（速いワイヤー）/ R（実 DS 実描画）/ **H（ハイブリッド=推奨）** のどれで作るか。
2. **参照パターン**（Mobbin 等を入口に見せる）を入れるか（"見て考える"を強化するが任意）。
3. 段階導入の**①（体験骨子）から着工**でよいか（最小工数で"相談できない"を即解消）。

## 7.5 Design 成果物のフォルダリング＆命名規約（DEC-054 で確定）

> CEO:「Design は **repo の技術スタックに影響せず**に作れるように。`designidea-01.html` 的に**どんどん増やしていく**。**適切なフォルダリング規約**を設けて。」

- **場所**：`<issue>/design/` — **1 Issue = 1 design フォルダ**。その Issue の全デザインアイデアを蓄積（現ストア＝`.bezier/drafts/<ulid>-<slug>/design/`、gitignore ローカル）。
- **ファイル名**：**`NN-<kebab-slug>.html`**
  - `NN` = 2 桁ゼロ埋めの**連番**（01, 02, 03…）。**使い回さない＝増えていく**（round1 = 01–03 / round2 = 04–05…）。
  - `<slug>` = 方向の短い kebab 名（英小文字ハイフン）。例 `01-toolbar-filter.html` / `02-column-header-menu.html` / `03-compact-cards.html`。
- **中身＝スタック非依存の自己完結 HTML**：**インライン CSS のみ**。repo のフレームワーク・コンポーネント・既存コードを**読まない／真似ない／依存しない**。Spec から自由にビジュアルの方向を出す（実装制約は Build の仕事）。`sandbox=""` で安全描画。
- **メタ**（ファイル内）：`<title>` = 名前、`<!-- bezier:prompt: … -->` = 方向の一言。slug/title が board のラベル。
- **参照**：`@01` `@02`（番号）で特定アイデアを指して相談・展開。
- **棲み分け**：Design（スタック非依存・自由・蓄積）↔ Build（実スタックで実装・実描画）。橋渡し＝「この案で進める」。
- **参照ソース（Mobbin 等）はユーザーに委ねる**（DEC-054）：エージェント＝ユーザー自身の Claude Code なので、ユーザーの参照 MCP / CLAUDE.md のデザイン指針が**そのまま継承**される。Bezier は特定ツールを前提にせず「あれば使う」だけ（[[bezier-inherits-repo-conventions-moat]]）。

---

## 8. 一言要約

> 今の Design は「**実 DS に偽装した崩れる静的 HTML を、会話を殺して 1 枚ずつ生成し、眺めるだけ**」。これを「**チャットの続きとして複数方向を安く出し（ワイヤー）、案を指して相談し、採用候補だけ実 DS で確認して決める**」に変える。＝**board × chat 連結 ＋ 発散/収束ラウンド ＋ fidelity の二段構え**。
