### DF-1 — “Untitled” をやめる（Issue 詳細ヘッダ＋サイドバー）  ｜ S ｜ 判断不要 ｜ ☐

## C. 再設計・プロンプト（大・提案→合意→着手）

### DF-8 — Spec に書く中身の見直し（ログ分離 / レポート化）  ｜ M–L ｜ 要提案 ｜ ☐
- CEO「**ログ系が Spec にあるのは良くない**（太ると LLM 出力が甘くなる）。ログは専用 md へ。Spec に**追加され（るよう誘導され）る内容を見直し**、他 md/html への分割を提案して。調査・検討は**レポート化**してほしい（参考サイト分析は◎、フォント検討は md なしで 3–4 案だけだった＝読んで判断したい）」。
- **私の案**: Spec＝**耐久する決定/受け入れ基準のみ**。ログ→`.bezier/` 活動ログ or `notes/`。調査→`docs/research/` レポート md。プロンプト＋ doc テンプレを棚卸しして「何を Spec に入れ、何を別ファイルにするか」を表で提案。

### DF-2 — Design タブの HTML の役割を再定義  ｜ L ｜ 要提案＋CEO 判断 ｜ ☐
- CEO の意図: HTML＝「1 つのデザイン案」ではなく「**md より html が都合よいケースに生成されるもの**」。自由度高く、**3 案の FV を 1 つの html に並べて比較**もあり。MD/HTML の判断は LLM 任せ。→「html を作ってデザイン確定」という前提が崩れるので **prompt 等を調整**。「Adopt this」ボタンは廃し **chat で誘導**。HTML ヘッダも **md 同様ファイル名表示**。
- **CEO からの問い**: html のパターン/デザインルールは一定持つべき？ 無くて良い？
  - **私の回答（推奨）**: **軽いルールは持つ**べき。固定 1 パターンに縛らないが、`design-system.md`/トークンを**コンテキストとして継承**させ、多パターン html でも**ブランドが破綻しない最低限**だけ効かせる。「1 html=1 デザイン決定」制約と Adopt は外す。＝“枠は外すがトーンの土台は残す”。

### DF-5 — Share の選択モデルを今の設計に合わせる  ｜ L ｜ 要提案 ｜ ☐
- CEO「選び方が今の設計とズレた。**Design のどの資料／Prototype の Preview・Map・QA のどれを含むか**を、生成済みタブ配下に対して**共有する/しない**で選ぶ。既定=全部、設定で減らせる（**checkbox を外す**）。受け手側も Spec が必ずトップとは限らない＝**作り手の Issue 詳細 UI と同じ Segmented Control＋タブ**で。共有が無いセグメントは受け手側で**丸ごと出さない**」。
- **私の案**: 賛成。DEC-094/099–102 の共有フローを、この「セグメント×タブの取捨」モデルに再設計。受け手ページを作り手 UI のミラーに。
