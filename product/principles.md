# continuum — プロダクト原則 / the maker-loop

## continuum は何か

**一人の「maker」が Spec → Design → Mock → QA を連続的に回す単一の作業面。**
既存プロダクトの文法を読み込み、その実パーツを流用してモックを生成・編集し、仕様と QA まで地続きにする。

## the maker loop（中核モデル）= 仕様駆動ループ（spec-driven）

参照: Amazon **Kiro**（spec → tasks → 実装 のAIコーディング）。continuum はこの「仕様駆動」を、デザイン/モックの**発散・収束**まで含めて一気通貫にする。

```
Intent ─→ Spec(下書き) ─→ Design ─→ Mock ─→ Spec(確定) ─→ QA ─→ Handoff(spec→タスク) ─→ [AI実装]
                            │          │          ▲
                            └ 発散→収束(discover) ┘ 結果をSpecに反映
```

CEO の5ステップに対応:
1. 仕様を適切に作成 → **Spec（下書き）**
2. デザイン実装してモック → **Design / Mock**
3. パターン追加・discover・発散↔収束 → **Mock のディスカバリーボード**
4. スペックをファイナライズ → **Spec（確定）**（Mockの収束結果を差分反映）
5. QAケース洗い出し → ハンズオフ → **QA / Handoff**（spec から実装タスクへ自動分解）
→ その先: エンジニアが AI と実装（**一気通貫**で繋ぐのが将来像）

- 各段階は別ツール・別役割に分断されず、同じデータ（spec / scene-graph / DS / tasks）の上で連続する。
- **Mock は発散の場**（パターンを広げる）、**収束で1案採用** → それが Spec を更新する。直線でなくループ。
- AI は全段階に同席する（書く・作る・検証する・分解する）が、**判断は maker が持つ**。

## continuum は何で「ない」か（禁則）

- ❌ **白紙生成ツールではない**。既存の文脈を無視した「それっぽいUI」は作らない。
- ❌ **コード生成器ではない**。コードは scene-graph からの downstream projection（export）。真実ではない。
- ❌ **役割分業の効率化ツールではない**。PdM↔Designer の往復を速くするのでなく、往復そのものを消す。
- ❌ **Figma/Notion の劣化版ではない**。既存ツールに対する微差別化（「少しUIが良い」）は狙わない（固い問題の禁則）。

## 設計の中核決定（不可逆に近い）

1. **モックの真実 = scene-graph（`mock_nodes` ツリー）**。コードblobでもvector shapeでもない。葉ノード = repoの実コンポーネントのinstance。
2. **既存を置換せず、上に立つ**（Sierra SoA）。既存 DS / repo を読み参照するが壊さない。
3. **build ≠ 検証**。プロトタイプは検証ではない。dogfood と行動データが検証。
4. **DS 忠実度は機能でなく前提**。生成物が既存デザインシステムを尊重しないなら、それは欠陥。

## 品質バー（QA ゲートの北極星）

- 生成モックを見た maker が「**これは自分のプロダクトだ。続きを作れる**」と言えるか。
- 既存パーツの流用が **正しい**（prop schema 準拠・DS 準拠）か。
- 編集が **可逆・構造的**（テキスト差分でなくツリー操作）か。
